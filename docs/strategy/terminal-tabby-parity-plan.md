# 终端体验对齐 Tabby 的方案

> 2026-06-03 | 目标：在现有 xterm.js + PTY 架构上，把终端体验拉近 Tabby，优先保证稳定性，不牺牲交互上下文。

---

## 一、目标

现有终端已经可用，但还没有到 Tabby 那种“顺滑、稳定、不中断上下文”的程度。当前主要问题是：

- 字符流解码不够稳，容易出现乱码
- 字体调整会触发终端重建，导致 Codex / Claude 这类 TUI 上下文丢失
- 启动和发送命令前有固定等待，体感偏慢
- 搜索句柄的生命周期不够稳，存在失效风险

这个方案的目标不是重做终端，而是在当前架构上把体验补齐。

---

## 二、推荐方案

### 1. 终端实例永不重建

- 每个 tab 的 xterm 只初始化一次
- 字体、主题、大小都只更新现有实例的 `options`
- 字体变化后只 `fit()`，不 remount、不清屏、不 replay

**收益**：避免黑屏、避免交互式 TUI 上下文丢失  
**成本**：低

### 2. PTY 输出做增量 UTF-8 解码

- 后端不要对每个 chunk 直接 `from_utf8_lossy`
- 改成带缓存的增量解码，保留半个字符的尾部字节
- 只在字节完整时再发给前端

**收益**：消除中文、emoji、框线字符跨包乱码  
**成本**：中

### 3. 去掉硬编码等待，改为就绪信号驱动

- 不再依赖 200ms / 800ms 这种固定等待
- 改成 `onReady + first fit + shell-ready` 驱动
- 命令发送只保留最小必要等待

**收益**：启动和交互更快，体验更接近 Tabby  
**成本**：中

### 4. 搜索句柄改成稳定 getter

- 不直接暴露可能失效的快照引用
- 统一通过 getter 取当前 addon 实例

**收益**：终端内搜索稳定  
**成本**：低

### 5. 渲染层保持默认 canvas，WebGL 作为可选项

- 默认继续用 canvas，保证稳定
- WebGL 只作为性能开关，不做默认

**收益**：兼顾稳定和性能  
**成本**：低

---

## 三、成本评估

| 项目 | 复杂度 | 风险 |
|------|--------|------|
| 永不重建终端实例 | 低 | 低 |
| 增量 UTF-8 解码 | 中 | 中 |
| 去硬等待、改就绪驱动 | 中 | 中 |
| 搜索句柄稳定化 | 低 | 低 |
| WebGL 可选开关 | 低 | 低 |

**总体判断**：这是“中等成本、低重构风险”的优化，不需要改架构，只是在关键链路上补正确性。

---

## 四、建议实施顺序

1. 先修 UTF-8 解码，消除乱码
2. 再修字体更新不重建，消除黑屏和上下文丢失
3. 然后收紧等待逻辑，改善体感速度
4. 最后补搜索和渲染开关，收尾体验

---

## 五、结论

如果目标是“接近 Tabby 的完成度”，最关键的不是加功能，而是：

- 不重建终端
- 不丢字节边界
- 不用硬等待替代就绪信号

这三件事做好后，终端体验会明显更稳。要到“完全像 Tabby”，还可以继续做 WebGL、渲染细节和 TUI 兼容性调优，但那是第二阶段。
